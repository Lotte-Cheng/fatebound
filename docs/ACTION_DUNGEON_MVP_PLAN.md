# Fatebound Action Dungeon MVP Plan v2

更新时间：2026-02-20

## 进度状态

- `Phase 2A（生存小关化）`：已完成并可玩。
- `Phase 2B（构筑系统闭环）`：进行中（已落地槽位与协同第一版）。
- `Phase 2C~2E`：待后续版本推进。

## 目标

在不推翻现有 `rules-first` 架构前提下，把当前地牢白盒升级为：

1. **每个小关为 Vampire Survivors 风格生存战斗**（定时波次 + 构筑选择）。
2. **神像交互升级为“读碑文 -> 仪式召神 -> 固定轮次对话 -> 规则结算”**。
3. **提供一条完整构筑路线与完整结局闭环**，可 10-15 分钟稳定体验。

## 不变原则

1. 所有数值变化由规则系统决定。
2. AI 只做意图解析 JSON 与叙事文本。
3. 叙事文本不得新增机制效果。
4. 同 seed + 同输入 + 同选择，结果可复现。

## 当前基线（已完成）

1. 动作地牢主循环可运行（移动、射击、清房、选路、锁门钥匙、祈福房）。
2. 小地图探索点亮与相邻房间有限预判。
3. 祈福房已有“清怪后 + 靠近神像 + 输入冻结”交互。
4. 规则层测试可运行（determinism / threshold / delayed curse）。

## 目标循环（V2）

1. 进入房间 -> 开始 45~90 秒生存战斗（自动刷怪）。
2. 战斗中掉落经验晶体，达到等级时给 3 选 1 构筑项。
3. 战斗结束后开启门/事件点。
4. 在神像房可先阅读碑文，再执行仪式召神。
5. 召神后进行固定轮次对话（建议 3 轮）。
6. 每轮对话输出意图 JSON -> 规则引擎判定是否给奖励/代价。
7. 达成结局条件后进入终局房，播放完整结局文本并给出结局标签。

## Phase 规划

## Phase 2A - 生存小关化（Vampire-like）

范围：
1. 将 `battle` 房改为计时生存模式（`room_timer_sec`、`spawn_curve`）。
2. 增加经验掉落与局内等级（`xp`, `level`, `xp_to_next`）。
3. 每次升级触发 3 选 1 构筑选择。
4. 房间通关条件改为“存活到计时结束”或“击败精英目标”。

数据新增建议：
1. `data/dungeon_layout.json`：`room_timer_sec`, `spawn_profile_id`。
2. `data/spawn_profiles.json`：波次、密度、精英点。
3. `data/build_nodes.json`：构筑节点池与权重。

验收：
1. 任意战斗房可稳定运行 60 秒生存。
2. 玩家在 1 房内至少触发 1 次升级选择。

当前落地结果（已完成）：
1. 战斗房已改为生存刷怪模式，支持 `spawn_total_limit` 上限与清场结算。
2. 难度曲线由 `data/spawn_profiles.json` 控制，按房间递增。
3. 教学房 `r01` 实装“操作引导 + 固定升级选项 + 刷怪量刚好升 1 级”。
4. 教学阶段提供世界空间箭头、Hint 脉冲、升级面板高亮等显性引导。
5. 战场区域改为动态避让左侧 HUD，避免遮挡左上角信息。

## Phase 2B - 构筑系统闭环

范围：
1. 建立 4 类构筑槽位：武器、被动、神赐、诅咒债务。
2. 构筑节点由规则系统下发，战斗层只执行效果。
3. 增加构筑协同标签（例如 `projectile`, `ritual`, `survival`）。

验收：
1. 形成可感知流派差异（输出/生存/召神收益）。
2. 同 seed 可复现构筑候选与最终结算。

当前落地结果（第一批）：
1. `data/build_nodes.json` 增加 4 类槽位：`weapon/passive/godsend/debt` 与 `slot_limits`。
2. 构筑节点增加 `tags`，用于协同倾向抽取。
3. 增加 `synergy_rules`：满足标签组合后激活一次性协同效果（由构筑系统应用并记录日志）。
4. 升级面板与状态栏显示构筑槽位占用、主标签与已激活协同数量。
5. 增加 `build_progression`：`tier_cost`、每级预算增长与前置标签（`required_tags`）约束。

补充（战斗体验增强）：
1. 怪物死亡掉落经验晶体，角色拾取后累积经验条。
2. 新增 `spawn_profiles.global_tuning`，可一处调节怪物数值、刷怪频率与后期怪海强度。

## Phase 2C - 神像到神明直连交互

范围：
1. 神像旁常驻碑文（可阅读神像背景与偏好）。
2. 增加仪式资源与仪式按钮（例如 `sigil`, `blood`, `fate` 消耗）。
3. 仪式成功后出现神明实体对话界面。
4. 固定对话轮次（建议 3 轮，可配置）。
5. 每轮可获得奖励机会，但由规则判定概率和强度。

数据新增建议：
1. `data/statues.json`：碑文、仪式条件、关联神明。
2. `data/gods.json`：人格、禁忌、偏好、语气、奖励池偏置。
3. `data/dialogue_config.json`：`max_turns`, `reward_chance_curve`, `cooldown`。

验收：
1. 玩家可完成“读碑文 -> 仪式 -> 对话 3 轮 -> 结算”。
2. 对话文本风格符合神明人设且不越权改数值。

## Phase 2D - AI 接入（可选在线）

范围：
1. 先保留本地 stub（离线可跑）。
2. 增加 GPT API Provider 适配层（可切换 `stub/openai`）。
3. 严格 JSON schema 校验（意图解析）与内容白名单（叙事）。
4. 对话建议系统：每轮给 2-3 条“符合人设且高收益”的建议表达。

验收：
1. 断网可回退 stub。
2. 在线模型异常不阻塞主流程。
3. 意图 JSON 校验失败时自动重试或降级模板。

## Phase 2E - 完整构筑路线与结局落地

范围：
1. 固定一条推荐构筑路线（从 1 房到终局）。
2. 完成至少 1 个完整主结局文本与触发条件。
3. 增加失败结局兜底（死亡/仪式崩坏）。

验收：
1. 10-15 分钟内可稳定跑完整路线与结局。
2. 结局依据构筑与对话选择，而非随机文本。

## 技术接口约束（关键）

1. AI 意图接口输出：
`wish_type`, `tone`, `risk_preference`, `constraints[]`, `target`, `ritual_action`。
2. 规则输出接口：
`reward_ids[]`, `curse_ids[]`, `status_effects[]`, `narrative_seed`, `delta_preview`。
3. 神明叙事接口：只允许复述规则结果，不得新增效果条目。

## 风险与对策

1. 风险：VS 生存战斗引入高频事件，规则调用开销大。
   对策：战斗内事件做批处理，仅在关键事件点提交规则层。
2. 风险：AI 对话偏离人设。
   对策：人格约束 + forbidden topics + 失败回退模板。
3. 风险：构筑过强导致无脑通关。
   对策：为节点添加 `tier_budget` 与房间难度曲线联动。

## 下一步执行顺序（开发顺序）

1. 先做 `Phase 2A`（小关生存化）
2. 再做 `Phase 2C`（神像->仪式->召神）
3. 然后接 `Phase 2D`（API Provider）
4. 最后完成 `Phase 2E`（完整路线与结局）
